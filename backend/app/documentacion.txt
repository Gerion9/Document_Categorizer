DOCUMENTACION DE CAMBIOS - INTEGRACION OCR + PINECONE + GEMINI

Proyecto:
Document_Categorizer

Ubicacion de esta documentacion:
backend/app/documentacion.txt


1. OBJETIVO DE LOS CAMBIOS

Se implemento una capa de indexacion semantica con Gemini + Pinecone dentro del
backend Python de Document_Categorizer para que el flujo actual del proyecto
pueda:

- extraer texto OCR de paginas
- dividir ese texto en chunks utiles
- generar embeddings semanticos
- guardar esos embeddings en Pinecone
- consultar Pinecone por caso, pagina, seccion y checklist
- enriquecer la verificacion AI del sistema QC
- mantener sincronizado el indice cuando el usuario reorganiza paginas

La idea no fue reemplazar la logica principal del sistema, sino integrarla al
flujo real ya documentado en README:

- subir archivos
- dividir PDFs en paginas
- clasificar paginas en secciones
- extraer OCR
- verificar checklist y QC
- exportar resultados


2. ARQUITECTURA GENERAL AGREGADA

Se agregaron nuevos servicios al backend:

- services/rag_config.py
- services/pinecone_client.py
- services/embedding_service.py
- services/chunking_service.py
- services/ocr_index_service.py
- services/checklist_index_service.py
- services/retrieval_service.py
- services/indexing_service.py

Estos servicios trabajan asi:

1. extraction_service.py extrae OCR con Gemini.
2. indexing_service.py coordina la indexacion de una pagina ya extraida.
3. chunking_service.py divide el OCR en chunks.
4. embedding_service.py genera embeddings con Gemini.
5. pinecone_client.py abre el indice Pinecone.
6. ocr_index_service.py hace upsert/query de chunks OCR.
7. checklist_index_service.py indexa respuestas QC.
8. retrieval_service.py compone consultas semanticas reutilizables.

6. OCR Y EXTRACCION

Archivo involucrado:

- routers/extraction.py
- services/extraction_service.py
- services/indexing_service.py

Comportamiento actual:

- El endpoint de extraccion sigue funcionando igual para el usuario.
- Cuando se extrae texto de una pagina, el backend intenta indexarla en
  Pinecone automaticamente.
- Si Pinecone o embeddings no estan configurados, la extraccion sigue
  funcionando y la indexacion queda como "skipped".
- Si Pinecone falla, no se pierde el OCR ya extraido.

Endpoints relacionados:

- GET /api/extraction/status
- POST /api/pages/{page_id}/extract
- GET /api/pages/{page_id}/extraction
- POST /api/cases/{case_id}/extract-batch
- POST /api/pages/{page_id}/reindex
- POST /api/cases/{case_id}/reindex
- POST /api/cases/{case_id}/rag/query

8. PINECONE IMPLEMENTADO

Archivos:

- services/pinecone_client.py
- services/ocr_index_service.py
- services/checklist_index_service.py

Buenas practicas aplicadas:

- namespace por caso usando case_id

Esto mejora:

12. SINCRONIZACION CON EL FLUJO REAL DEL PROYECTO

Este fue uno de los cambios mas importantes.

Problema anterior:

- Pinecone se llenaba cuando se extraia OCR
- pero si el usuario movia paginas entre secciones, cambiaba la primaria,
  agregaba referencias secundarias o modificaba codigos de secciones, la
  metadata indexada podia quedar vieja

Solucion implementada:

Archivo:

- routers/pages.py
- routers/documents.py

Ahora Pinecone se resincroniza cuando:

- se clasifica una pagina
- se desclasifica una pagina
- se marca como extra
- se agrega un section link
- se quita un section link
- se cambia la seccion primaria
- se elimina una pagina
- se cambia el code de un DocumentType
- se cambia name/code/parent de una Section

Tambien se recalculan correctamente:

- path_code
- depth
- subarboles de secciones


13. SECTION LINKS Y QC

El sistema actual usa page_section_links para referencias secundarias.

Antes:

- parte del QC buscaba paginas por page.section_id

Ahora:

- tambien usa PageSectionLink
- esto permite que una pagina vinculada como referencia secundaria tambien sea
  considerada por retrieval y verificacion

Esto alinea Pinecone con el comportamiento real del tab Organizar.

15. LO QUE YA FUNCIONA MEJOR AHORA

Escenario real del README:

1. Se suben PDFs o imagenes.
2. Se convierten a paginas.
3. Se clasifican en una taxonomia.
4. Se extrae OCR con Gemini.
5. Ese OCR se indexa en Pinecone.
6. Si luego cambian la organizacion del caso, Pinecone se actualiza.
7. El QC y el autopilot consumen OCR + imagen + contexto semantico.

Resultado:

- mejor consistencia entre UI, DB y Pinecone
- mejor retrieval por seccion
- mejor soporte para evidencias secundarias
- mejor estabilidad en respuestas AI


16. LIMITACIONES ACTUALES

Aunque la integracion ya es bastante solida, todavia hay cosas que no son nivel
infraestructura enterprise:

- se usa BackgroundTasks en vez de una cola durable tipo Celery o RQ
- no se implemento OCR PDF-native con Files API de Gemini
- no se implemento reranking posterior a Pinecone
- falta prueba end-to-end contra credenciales reales del usuario
- falta suite de tests automatizados para esta capa nueva

Esto significa que:

- el sistema deberia funcionar bien en el flujo local normal
- pero para alto volumen o produccion fuerte aun conviene endurecer mas


17. COMO PROBAR EL SISTEMA

Completar backend/.env con credenciales reales.

Instalar dependencias:

python -m venv venv
.\venv\Scripts\Activate.ps1
pip install -r requirements.txt

frontend:
npm install

Paso 3:
Iniciar backend:

uvicorn app.main:app --reload --port 8000

Paso 4:
Iniciar frontend:

npm run dev

Paso 5:
Verificar:

- http://localhost:8000/api/health
- http://localhost:8000/api/extraction/status

Paso 6:
Prueba funcional recomendada:

- crear un caso
- subir un PDF
- clasificar paginas en secciones
- extraer texto de 1 pagina
- mover esa pagina a otra seccion
- ejecutar reindexado o dejar que se resincronice
- correr AI verify o autopilot sobre un QC checklist


18. RESUMEN FINAL

La integracion hecha en este proyecto no solo agrega Pinecone, sino que lo
conecta con el flujo real de Document_Categorizer.

Lo mas importante que se logro:

- OCR indexable
- retrieval por caso y seccion
- QC con contexto semantico
- namespaces por caso
- salidas estructuradas en Gemini

En otras palabras:

Pinecone ya no esta "pegado aparte", sino conectado al flujo vivo del caso.
